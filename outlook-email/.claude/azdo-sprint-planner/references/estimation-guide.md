# Sprint 估算指南

本文檔提供完整的 Sprint 估算方法、技巧和最佳實踐。

## 估算基礎

### 什麼是故事點數（Story Points）？

故事點數是一個相對的估算單位，用來衡量工作項目的：
- **複雜度** - 技術難度
- **工作量** - 需要的時間和精力
- **不確定性** - 風險和未知因素

**重要原則**：
- 故事點數是**相對的**，不是絕對的時間
- 團隊內部需有一致的參考基準
- 隨著團隊經驗增長，估算會更準確

### 故事點數量表

#### 修改後的費氏數列（推薦）

| 點數 | 複雜度 | 參考工作量 | 範例 |
|------|--------|-----------|------|
| 1 | 極簡單 | < 2小時 | 修改文字、簡單配置變更 |
| 2 | 簡單 | 2-4小時 | 新增簡單欄位、基本 UI 調整 |
| 3 | 中等偏簡單 | 4-8小時 | 實作簡單 API、基本 CRUD 功能 |
| 5 | 中等 | 1-2天 | 實作完整功能、多個 API 端點 |
| 8 | 複雜 | 2-3天 | 複雜業務邏輯、多個組件整合 |
| 13 | 非常複雜 | 3-5天 | 架構變更、複雜整合 |
| 20 | 極度複雜 | > 5天 | 需要拆解成更小的項目 |
| 40 | Epic級別 | > 10天 | **必須拆解** |

| 大小 | 對應點數 | 說明 |
|------|----------|------|
| XS | 1-2 | 極小的變更 |
| S | 3-5 | 小型功能 |
| M | 5-8 | 中型功能 |
| L | 8-13 | 大型功能 |
| XL | 13-20 | 非常大的功能 |
| XXL | 20+ | 需要拆解為多個項目 |

## 估算技巧

### Planning Poker

**步驟**：

1. **選擇參考故事**
   - 團隊選擇一個大家熟悉的 PBI 作為基準（例如：3點或5點）
   - 所有後續估算都相對於這個基準

2. **討論工作項目**
   - Product Owner 說明需求
   - 團隊提問並釐清細節
   - 討論技術考量和風險

3. **獨立估算**
   - 每個成員私下選擇一個點數
   - 使用 Planning Poker 卡片或線上工具

4. **同時揭示**
   - 所有成員同時展示選擇的點數
   - 避免錨定效應（anchoring bias）

5. **討論差異**
   - 如果估算差異大（例如：3 vs 13），討論原因
   - 最高和最低估算者說明理由
   - 重點關注被忽略的複雜度或過度估算的部分

6. **重新估算**
   - 經過討論後再次估算
   - 重複直到達成共識（通常2-3輪）

**線上工具**：
- Azure DevOps Planning Poker 擴充功能
- PlanningPoker.com
- Scrum Poker Online

### 三點估算法

適合處理不確定性高的項目。

**公式**：
```
預期點數 = (樂觀估算 + 4×最可能估算 + 悲觀估算) / 6
```

**範例**：
```
樂觀情況：3 點（一切順利）
最可能：5 點（正常情況）
悲觀情況：13 點（遇到未知問題）

預期點數 = (3 + 4×5 + 13) / 6 = 36 / 6 = 6 點
```

### 比較估算法

1. 將所有 PBI 排列在牆上（或虛擬看板）
2. 團隊協作將它們按複雜度排序
3. 分組並賦予點數
4. 快速且視覺化

### 專家判斷法

- 適合技術複雜度高的項目
- 由專家提供估算，團隊驗證
- 注意避免過度依賴單一專家

## 拆解過大的項目

### 何時需要拆解？

**強烈建議拆解**：
- 故事點數 > 13
- 無法在一個 Sprint 內完成
- 團隊成員對估算差異大（如：3 vs 20）
- 包含多個獨立的價值交付點

### 拆解策略

#### 1. 按工作流程拆解

```
原始 PBI (20點): "實作完整的訂單管理系統"

拆解後:
├── PBI (5點): "實作訂單創建 API"
├── PBI (5點): "實作訂單查詢和列表 API"
├── PBI (3點): "實作訂單狀態更新功能"
├── PBI (3點): "實作訂單取消功能"
└── PBI (4點): "實作訂單歷史記錄"
```

#### 2. 按介面類型拆解

```
原始 PBI (15點): "實作使用者管理功能"

拆解後:
├── PBI (5點): "實作後端使用者 CRUD API"
├── PBI (5點): "實作前端使用者列表和詳情頁面"
└── PBI (5點): "實作使用者權限管理介面"
```

#### 3. 按技術層次拆解

```
原始 PBI (18點): "整合第三方支付系統"

拆解後:
├── PBI (3點): "研究和選擇支付供應商 API"
├── PBI (5點): "實作支付服務層"
├── PBI (5點): "整合支付 UI 組件"
└── PBI (5點): "實作支付回調和錯誤處理"
```

#### 4. 按使用情境拆解

```
原始 PBI (20點): "實作搜尋功能"

拆解後:
├── PBI (5點): "實作基本關鍵字搜尋"
├── PBI (5點): "實作進階篩選（日期、類別）"
├── PBI (5點): "實作搜尋結果排序"
└── PBI (5點): "實作搜尋歷史和建議"
```

#### 5. 按 MVP 和增強功能拆解

```
原始 PBI (15點): "實作報表產生器"

拆解後:
├── PBI (5點): "實作基本 PDF 報表產生 (MVP)"
├── PBI (3點): "新增報表範本選擇"
├── PBI (3點): "新增圖表視覺化"
└── PBI (4點): "新增報表排程和自動發送"
```

## 速度（Velocity）追蹤

### 計算團隊速度

**速度** = 每個 Sprint 完成的故事點數總和

#### 範例

| Sprint | 承諾點數 | 完成點數 | 備註 |
|--------|----------|----------|------|
| Sprint 1 | 25 | 20 | 團隊磨合期，有些估算不準 |
| Sprint 2 | 28 | 25 | 改善了工作流程 |
| Sprint 3 | 30 | 28 | 一個 PBI 移至下個 Sprint |
| Sprint 4 | 28 | 28 | 達到穩定速度 |
| **平均速度** | - | **25.25** | 取最近 3-4 個 Sprint 平均 |

### 使用速度規劃

**規劃下個 Sprint**：
```
團隊速度：25 點
可用容量：考慮假期和請假後，調整為 22 點
建議承諾：20-22 點的 PBI
```

### 速度趨勢分析

```python
# 速度趨勢計算
sprints = [
    {"sprint": 1, "velocity": 20},
    {"sprint": 2, "velocity": 25},
    {"sprint": 3, "velocity": 28},
    {"sprint": 4, "velocity": 28},
    {"sprint": 5, "velocity": 26},
]

# 計算移動平均（最近3個Sprint）
def moving_average(data, window=3):
    return [
        sum([s["velocity"] for s in data[max(0, i-window+1):i+1]]) / min(i+1, window)
        for i in range(len(data))
    ]

# 預測下個Sprint
recent_avg = sum([s["velocity"] for s in sprints[-3:]]) / 3
print(f"預測速度: {recent_avg:.1f} 點")  # 輸出: 27.3 點
```

## 歷史資料使用

### 建立估算參考庫

維護已完成項目的資料庫：

```markdown
## 參考 PBI 庫

### 3 點範例
- **簡單 REST API 端點**
  - 實際完成時間：4 小時
  - 技術：FastAPI, SQLAlchemy
  
- **基本 UI 表單**
  - 實際完成時間：6 小時
  - 技術：React, Formik

### 5 點範例
- **完整 CRUD 功能**
  - 實際完成時間：1.5 天
  - 包含：API + 前端 +測試
  
- **第三方 API 整合**
  - 實際完成時間：2 天
  - 包含：封裝層 + 錯誤處理

### 8 點範例
- **複雜報表功能**
  - 實際完成時間：3 天
  - 包含：查詢優化 + 資料聚合 + 視覺化
```

### 類比估算

使用歷史資料進行類比：

```
新 PBI: "實作產品評論系統"

尋找類似的歷史項目:
- "實作留言系統" (過去 Sprint 3, 8點, 3天完成)
- 相似度: 80%
- 差異: 新項目需要評分功能 (+30% 複雜度)

估算: 8 點 × 1.3 = 10.4 點 → 調整為 13 點
```

## 處理不確定性

### Spike（研究任務）

當團隊對技術不確定時：

```markdown
## Spike 範例

**標題**: SPIKE: 評估 WebSocket vs Server-Sent Events

**目標**:
- 比較兩種技術的優缺點
- 建立簡單的 POC
- 提供技術建議報告

**Time-box**: 2 天（固定，不論結果）

**故事點數**: 3 點

**產出**:
- 技術比較文檔
- POC 程式碼
- 實作建議

**注意**: Spike 完成後，再估算實際實作的PBI
```

### 不確定性緩衝

對於高風險項目，加入緩衝：

| 風險等級 | 緩衝係數 | 說明 |
|----------|----------|------|
| 低 | 1.0 | 熟悉的技術和領域 |
| 中 | 1.3 | 部分新技術或複雜整合 |
| 高 | 1.5-2.0 | 全新技術或架構變更 |

**範例**：
```
基礎估算: 5 點
風險等級: 高（新框架）
調整後: 5 × 1.5 = 7.5 → 8 點
```

## 估算檢查清單

在估算每個 PBI 前，確認：

- [ ] **需求清晰** - 驗收條件明確定義
- [ ] **技術可行** - 沒有未知的技術障礙（或已建立 Spike）
- [ ] **依賴識別** - 知道所有外部依賴
- [ ] **測試考慮** - 包含測試工作量
- [ ] **文檔考慮** - 包含必要的文檔工作
- [ ] **團隊共識** - 所有成員理解並同意估算
- [ ] **合理大小** - 可在一個 Sprint 內完成

## 常見估算錯誤

### 1. 錨定效應

**問題**: 第一個提出的估算影響其他人的判斷

**解決**: 使用 Planning Poker 同時揭示估算

### 2. 忽略測試和文檔

**問題**: 只估算開發時間，忘記測試和文檔

**解決**: 明確定義 "Done" 的標準，包含測試和文檔

### 3. 過度精確

**問題**: 嘗試將點數轉換為精確的小時數

**解決**: 記住點數是相對的，不是絕對時間

### 4. 個人能力差異

**問題**: 基於特定個人的能力估算

**解決**: 基於團隊平均能力估算

### 5. 理想時間 vs 實際時間

**問題**: 只考慮純開發時間，忽略會議、中斷等

**解決**: 故事點數應反映實際工作量，包含所有干擾

## 估算改善

### 估算準確度追蹤

```python
# 追蹤估算 vs 實際
estimation_tracking = [
    {
        "pbi_id": 12345,
        "title": "實作使用者認證",
        "estimated": 5,
        "actual": 5,
        "variance": 0
    },
    {
        "pbi_id": 12346,
        "title": "實作訂單列表",
        "estimated": 3,
        "actual": 5,
        "variance": +2  # 低估了
    },
    {
        "pbi_id": 12347,
        "title": "優化資料庫查詢",
        "estimated": 8,
        "actual": 5,
        "variance": -3  # 高估了
    }
]

# 計算準確度
total_estimated = sum([p["estimated"] for p in estimation_tracking])
total_actual = sum([p["actual"] for p in estimation_tracking])
accuracy = (total_actual / total_estimated) * 100

print(f"整體估算準確度: {accuracy:.1f}%")
# 輸出: 93.8%（16實際 vs 16估算）

# 識別模式
overestimated = [p for p in estimation_tracking if p["variance"] < 0]
underestimated = [p for p in estimation_tracking if p["variance"] > 0]
```

### 回顧和調整

在 Sprint Retrospective 討論：

1. **估算準確度** - 哪些項目估算準確/不準確？
2. **原因分析** - 為什麼會有偏差？
3. **學習點** - 下次如何改進？
4. **更新參考** - 調整估算參考庫

### 校準練習

定期進行估算校準：

```markdown
## 校準會議議程（每季度）

1. **回顧歷史估算**（30分鐘）
   - 檢視過去 3 個月的估算準確度
   - 識別模式和趨勢

2. **重新估算練習**（45分鐘）
   - 選擇 5-10 個已完成的 PBI
   - 團隊重新估算（不看原估算）
   - 比較新舊估算，討論差異

3. **更新基準**（15分鐘）
   - 確認或調整參考 PBI
   - 更新估算指南

4. **行動項目**（10分鐘）
   - 確定改善措施
```
